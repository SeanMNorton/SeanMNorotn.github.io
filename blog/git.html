Git Hub and Version Control

1.24.16

A Very Basic Overview

The command line, git and GitHub are the staples of coding and version control. GitHub is a repository hosting service, meaning it can hold all of our files and folders when coding. It’s great interface also makes it easy for managing and sharing projects/repos a.k.a. repositories. The command line matched with git is how we communicate to GitHub.

Using CLI (command line) we can “git clone” or copy our repos to our computer. Doing this makes sure that we can edit our files offline, we can also get to them from anywhere, by cloning/pulling them to any computer. When you make an edit on the file locally, it does not automatically update online, you must tell it to do so, this helps prevent from sharing broken code. Another safety precaution we can use is to edit on a new branch. A good way to see branches are as versions of the master(trunk) code where you branch off with new code that hasn’t been tested yet. This prevents breaking the master file that other people could be working on as well. Once your code (branch) is properly tested and works you can merge it with the master code. This whole process is part of version control.

Version control is a way that people keep track of changes made to their code, each update we commit to GitHub, GItHub keeps track of. This again helps make sure that when/if the code breaks we can go back to a past version that worked and fix any issues that seemed to have arrived in our code.
