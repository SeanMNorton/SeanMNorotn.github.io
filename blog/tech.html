SQL Injection:

“SQL injection is a code injection technique used to attack data-driven applications.” - wikipedia

These types of attacks are rather common, If a site is not protected than string literal escape characters can used as user input and change the output of an SQL statement.

This sounds a little confusing, but it’s rather straightforward, say you were on a website and forgot your password, you might click forgot password and then it would send you in an email your user_name and password. It would be getting this information from a database of some sort. One way we could pull this information would be to use SQL

SELECT password, user_name FROM users WHERE user_name = userName ;

This would return your username and password if you gave it your “userName” An attacker could use this opportunity to have their userName be ’ ’ OR ‘1’ = ‘1’. Let’s see how this could be a problem.

SELECT password, user_name FROM users WHERE name = ‘ ‘ OR ‘1’ = ‘1’;

In this case no user name will be an empty string, but ‘1’ will always = ‘1’ thus sending every username and password to the hacker.

There are quite a few different ways that this can be done as well, this was just one of the easier examples.

A way to protect against such an attack, specifically the one provided, would be to use parameterized queries. Also known as prepared statements. This way of coding allows your database to tell the difference between code and data/input data. In this way, if an attacker were to put OR 1 = 1 it would look for a string that matched the userName OR 1 = 1 instead of running it as code.
